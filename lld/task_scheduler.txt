# **Task Scheduler – Staff-Level Summary**

## **1️⃣ OOP Design**

* **Task**

  * Represents a single job
  * Attributes: `id`, `run_at`, `priority`, `block`, `status`, `retries`
  * Encapsulates execution logic

* **TaskQueue**

  * Thread-safe storage of tasks (`Mutex`)
  * Sorted by `run_at` and `priority`
  * Methods: `add`, `pop_ready`, `cancel`, `list`

* **TaskScheduler**

  * Orchestrates scheduling, cancellation, and execution
  * Maintains worker threads (single/multi-threaded)
  * Delegates ready tasks to execution

* **Worker Threads**

  * Consume ready tasks
  * Execute task blocks
  * Update task status (`completed`, `failed`)

* **Custom Exceptions**

  * `TaskAlreadyExistsError`, `TaskNotFoundError`, `TaskExecutionError`, `TaskCancelledError`
  * Improves clarity, makes error handling explicit

---

## **2️⃣ Edge Cases & Handling**

| Edge Case                   | Handling                                             |
| --------------------------- | ---------------------------------------------------- |
| Task scheduled in the past  | Executes immediately on next scheduler tick          |
| Duplicate task              | Raise `TaskAlreadyExistsError`                       |
| Cancel non-existent task    | Raise `TaskNotFoundError`                            |
| Task fails during execution | Status = `failed`, retries incremented, error logged |
| Concurrency                 | Thread-safe queue (`Mutex`) prevents race conditions |
| Recurring tasks             | Can extend Task with interval and auto-reschedule    |

---

## **3️⃣ Production-Level Approach**

**Scalability & Reliability:**

1. **Worker Pool**

   * Multi-threaded or multi-process execution
   * Handles CPU-bound or IO-bound tasks concurrently

2. **Persistent Storage**

   * In-memory queue replaced/augmented by Redis, Kafka, or DB
   * Tasks survive restarts or crashes

3. **Distributed Scheduling**

   * Multiple scheduler instances
   * Leader election or distributed queue for task coordination
   * Idempotent task execution to prevent duplicates

4. **Retry & Failure Handling**

   * Retries configurable per task
   * Dead-letter queue for permanently failed tasks

5. **Priorities & Timing**

   * TaskQueue maintains sorted order by `run_at` and `priority`
   * Allows urgent tasks to preempt pending lower-priority tasks

6. **Monitoring & Observability**

   * Metrics: pending, running, completed, failed tasks
   * Logging and alerting for SLA / latency violations

---

## **4️⃣ Staff-Level Talking Points**

> “The Task Scheduler is modular with clear separation: `Task` encapsulates jobs, `TaskQueue` handles storage and thread safety, and `TaskScheduler` orchestrates execution.
> I’ve used custom exceptions for clarity and robust error handling.
> For production, we can scale via worker pools, distributed queues, persistent storage, and monitoring.
> Concurrency, retries, idempotency, and task priority are all considered to handle real-world edge cases.”

# **Legend / Flow Explanation**

1. **TaskScheduler**

   * Public interface to **schedule, cancel, list tasks**
   * Maintains a reference to **TaskQueue**
   * `start` spins up worker thread(s) to process ready tasks

2. **TaskQueue**

   * **Thread-safe** queue (`Mutex`)
   * Tasks sorted by `run_at` and `priority`
   * Methods: `add`, `pop_ready`, `cancel`, `list`

3. **Task**

   * Represents a single job
   * Stores **id, run_at, priority, block, status, retries**
   * Can execute its block safely, handles exceptions

4. **Worker Threads**

   * Scheduler loop continuously calls `pop_ready`
   * Executes tasks concurrently (single-thread in demo, multi-threaded in production)

5. **Execution Flow**

   * `TaskScheduler` → `TaskQueue.pop_ready` → ready `Task` → execute block → update `status`
